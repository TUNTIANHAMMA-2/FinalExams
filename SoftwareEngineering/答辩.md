# 宿舍报修管理系统答辩

## 选题说明

本项目选择“宿舍报修管理系统”作为软件工程导论期末考查题目。该系统面向高校宿舍维修管理场景，解决学生报修渠道分散、维修进度不透明、派工记录难追踪、后勤统计依赖人工整理等问题。

## 系统目标

系统围绕“学生提交报修、管理员审核派工、维修员处理反馈、学生确认评价、后勤统计分析”形成完整业务闭环。核心目标是提高报修处理效率，规范维修流程，并为后勤部门提供可查询、可统计的数据依据。

## 需求分析要点

主要参与者包括学生、管理员、维修员和后勤管理部门。核心数据对象包括学生、宿舍、报修工单、派工记录、维修员和管理员。系统功能包括报修申请管理、工单审核与派工、维修处理与反馈、统计查询与导出。

## 设计说明

需求阶段使用 UML 类图描述系统静态结构，使用顶层 DFD 和 0 层 DFD 描述系统数据流，使用状态图描述报修工单的生命周期。总体设计阶段将 0 层 DFD 中的主要加工转换为系统功能结构图。详细设计阶段选择报修申请受理模块和派工处理模块，分别使用程序流程图和 N-S 图描述模块内部逻辑。

## 编码与测试说明

程序编码部分采用 PDL 伪代码描述报修申请受理模块，覆盖输入校验、宿舍档案匹配、附件检查、紧急程度判断、工单生成和管理员通知。测试部分针对该模块设计白盒测试和黑盒测试，白盒测试覆盖语句覆盖、判定-条件覆盖、条件组合覆盖和独立路径覆盖，黑盒测试使用等价类划分和边界值分析。

## 关键答辩问题

1. 为什么选择宿舍报修管理系统？

   该系统属于典型中小型事务管理信息系统，业务流程清晰，能够完整体现需求分析、总体设计、详细设计、编码和测试各阶段的软件工程方法。

2. 为什么选择增量模型？

   宿舍报修主流程相对明确，但后续可以逐步扩展移动端提醒、满意度统计、维修材料管理等功能。增量模型适合先交付核心流程，再按实际反馈扩展功能。

3. DFD 如何转换为系统结构图？

   0 层 DFD 中的报修申请受理、工单审核、派工处理、维修反馈确认、统计查询分别对应系统一级功能模块，数据存储通过模块接口统一访问。

4. 测试为什么选择报修申请受理模块？

   该模块包含必填项校验、档案匹配、附件循环检查和紧急故障判断，既有顺序结构，也有分支和循环，适合同时进行白盒测试和黑盒测试。

5. 如何保证前后设计一致？

   报告中的类图、DFD、结构图、详细设计、PDL 伪代码和测试用例均围绕“报修工单”及其处理流程展开，核心对象、功能模块和测试对象保持一致。
